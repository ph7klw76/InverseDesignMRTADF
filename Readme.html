XGBoost Ensemble (primary predictor)
Architecture: 20-member bootstrap ensemble
  • Each member uses a different random seed (42, 43, … 61)
  • Each member is trained on a bootstrap resample of the 176 molecules (sampling with replacement, same N)
  • subsample and colsample_bytree are independently drawn per member from Uniform(0.7, 0.9) and Uniform(0.6, 0.8)
  • 600 trees, depth 6, LR 0.05, L1=0.1, L2=1.0
Uncertainty = ensemble spread

For any new molecule: μ = mean of 20 predictions, σ = std of 20 predictions. No assumption about Gaussianity — purely empirical spread.

GPU acceleration: device="cuda", tree_method="hist"

The GPU histogram method on A100 is ~10× faster than CPU for building each tree. 5-fold CV runs first to validate hyperparams before ensemble training.

GPR comparison model
Matérn-5/2 kernel with white noise

C(1.0) × Matérn(ν=2.5, ℓ=ones(126)) + WhiteKernel(0.01)
The length-scale vector is optimised by gradient ascent on log-marginal-likelihood (5 restarts). Valid only because N=176 ≤ 250 (O(N³) scaling).

Why train both?

XGBoost provides faster inference on large candidate pools. GPR provides well-calibrated predictive uncertainty (coverage). Both are cross-validated independently.

Why SMILES, not Mol objects?

RDKit Mol objects cannot be reliably pickled across processes. Each worker calls MolFromSmiles() independently — cheap and pickle-free.

How each molecule is processed
def _compute_one_mol_fast(smi_and_names): mol = MolFromSmiles(smi) # ① RDKit 2D descriptors (~210 values) vals = _RDKIT_CALC.CalcDescriptors(mol) rdkit_dict = dict(zip(names, vals)) # ② Morgan r=2 → ExtFP bits (1024) ext_fp = AllChem.GetMorganFingerprintAsBitVect( mol, 2, nBits=1024) # ③ RDKit topological FP → FP bits (1024) std_fp = RDKFingerprint(mol, fpSize=1024) # ④ Morgan r=3 → GraphFP bits (1024) graph_fp = AllChem.GetMorganFingerprintAsBitVect( mol, 3, nBits=1024) # ⑤ Assemble 126 values in exact column order for k, name in enumerate(descriptor_names): if name.startswith("ExtFP"): out[k] = arr_ext[idx] elif name.startswith("GraphFP"): out[k] = arr_graph[idx] elif name.startswith("FP"): out[k] = arr_std[idx] else: out[k] = rdkit_dict[name]
Critical design decisions
_RDKIT_CALC built once at module level

MolecularDescriptorCalculator initialisation has overhead. Building it once per process (at import time) and reusing across all molecules in that worker saves ~0.5 s/mol overhead.

chunksize=64 for Pool.map

Sends 64 molecules to each worker at once instead of one-by-one. Reduces IPC overhead from O(N) messages to O(N/64), critical when N = 5000.

Column order guaranteed

Descriptors are assembled in exactly the same order as the training matrix columns (descriptor_names list from Step 1), so XGBoost features are correctly aligned.

Same imputer from Step 1

After parallel descriptor computation, the training-set median imputer (fitted on 176 molecules) fills any NaN cells — ensures the model sees the same feature distribution.

Constrained Expected Improvement formula
CEI(x) = EI(T₁−S₁) × P(T₂−S₁ < 0.20)
EI(x) = (y* − μ − ξ)Φ(Z) + σ φ(Z)
Z = (y* − μ − ξ) / σ
P(T₂−S₁ < 0.20) = Φ((0.20 − μ_T₂) / σ_T₂)
Variables
SymbolMeaning
y*Best T1−S1 in training set (minimum)
μ, σEnsemble mean & std for T1−S1
μ_T₂, σ_T₂Ensemble mean & std for T2−S1
ξ = 0.01Exploration bonus (jitter)
Φ, φNormal CDF and PDF
Why single-pass instead of 80 iterations?
Key insight: the ensemble is frozen

Traditional BO re-trains the surrogate after each evaluated point. Here the XGBoost/GPR ensemble is fixed — no wet-lab oracle to query. So CEI(x) is deterministic for every candidate, making 80 identical re-computations pure waste.

Single vectorised numpy pass

All 5000 candidates are scored simultaneously using numpy broadcasting. No Python for-loops. GPU handles the XGBoost inference, numpy handles the statistics. Total time <5 s regardless of N.

Ranking & Output
rank_candidates() — dual filter + lexicographic sort
numpy/pandas <1 s
›
Ranking logic (two gates + sort)
Gate 1 — Uncertainty filter

Keep only candidates where unc_T1_S1 < 0.10 eV. Rejects molecules the model is uncertain about — prevents reporting extrapolations as discoveries.

Gate 2 — Constraint filter

Keep only candidates where pred_T2_S1 < 0.20 eV. This is the hard MR-TADF requirement — T2 must lie close to S1 for RISC to be thermally accessible.

Sort: smallest T1−S1 first, tiebreak T2−S1

Primary objective is minimise T1−S1 (small singlet-triplet gap → efficient RISC). Secondary: smallest T2−S1 (tighter spin manifold).
